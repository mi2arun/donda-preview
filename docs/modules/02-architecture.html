System Architecture & Bidding-App Integration

A separate team builds the expo bidding mobile app; the Donda ERP is the single backend and system of record for all bidding and post-bidding data.

The Big Picture

The bidding app is a front-end only. It reads master data from the ERP and writes all expo activity (catalog, bids, wins, payments) back to the ERP through a defined API. The ERP then carries a won lot downstream through logistics, purchase, inventory and sales.

Boundary decision (confirmed): the ERP stores full expo data — the app keeps no separate database. The ERP team defines the API contract; the app team integrates against it.

Integration Principles

  • Single source of truth — no duplicated data store on the app side; the app is online-first.
  • Authentication — the app authenticates as a service client (OAuth2 client-credentials or signed API key); each bidder user is identified in the request.
  • Versioned API — base path /api/v1/…, backward-compatible evolution.
  • Idempotency — write endpoints accept an Idempotency-Key so flaky mobile connections don't create duplicates.
  • Offline tolerance — the app may queue actions offline and replay; idempotency + server IDs keep this safe.
  • Webhooks / polling — the ERP notifies the app of status changes, or the app polls updated_since.
