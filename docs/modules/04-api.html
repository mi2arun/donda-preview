Module 04 · API · ERP-defined

Integration API Specification

REST/JSON over HTTPS, base path /api/v1. All requests authenticated; all writes accept an Idempotency-Key. Timestamps are ISO-8601 UTC. This is the ERP-proposed baseline, to be finalised with the app team.

Read — Master Data (app pulls)

MethodEndpointPurpose
GET/group-companiesList Donda entities
GET/suppliersExpo/tender & third-party suppliers
GET/exposExpos/tenders (filter by status)
GET/mines · /originsMines & country origins
GET/currenciesCurrencies & exchange rates

Write / Read — Bidding Data (app pushes, ERP stores)

MethodEndpointPurpose
POST/exposCreate an expo/tender
POST/expos/{id}/lotsUpload lot catalog (bulk, ~600 lots)
GET/expos/{id}/lotsRetrieve catalog (updated_since)
POST/bidsSubmit a bid on a lot
PATCH/bids/{id}Update bid (withdraw / result)
POST/lots/{id}/resultMark Won/Lost (Won → creates Won Lot)
POST/lots/{id}/paymentRecord on-spot payment
GET/won-lotsApp reads won lots & downstream status

Stone Valuation & Media FROM APP

To receive what the lot-capture screen produces, the contract carries stone estimates and media. Stones may be embedded in the lot upload or sent separately.

MethodEndpointPurpose
POST/lots/{id}/stonesUpsert stone estimates (shape/color/clarity, rough wt, yield, price, pieces)
GET/lots/{id}/stonesRetrieve a lot's valuation breakdown
POST/stones/{id}/mediaUpload a photo/video for a stone (multipart)
PATCH/lots/{id}Set bid $/ct and (separate) lot number + description
Recommendation: the app should send lot_number and description as separate fields (the mockup currently combines them into one string). Confirm whether the full valuation breakdown persists in the ERP or only the final bid/ct.

Examples

Submit a bid

POST /api/v1/bids Authorization: Bearer <token> Idempotency-Key: 7c0e1f8a-... { "expo_lot_id": 4821, "bidder_user": "rahul", "bid_price_per_carat": 651.81, "bid_datetime": "2026-03-12T09:14:00Z", "remarks": "good colour, slightly included" } → 201 Created { "bid_id": 99213, "bid_status": "Submitted", "expo_lot_id": 4821 }

Mark a lot Won (creates Won Lot)

POST /api/v1/lots/4821/result { "result": "Won", "winning_bid_id": 99213, "win_date": "2026-03-12", "win_price_per_carat": 651.81, "payment": { "date": "2026-03-12", "reference": "TT-55129", "method": "Bank Transfer" } } → 200 OK { "lot_status": "Won", "won_lot_id": 1305, "won_lot_status": "Won" }

Upload a lot with stone estimates

POST /api/v1/expos/t1/lots { "lot_number": "47", "description": "FY+ SI1 24.45ct", "mining_origin": "CA-Diavik", "bid_per_carat": 3334, "stones": [ { "shape": "RD", "color": "FY", "clarity": "VS1", "rough_ct": 24.45, "yield_pct": 32, "price_per_ct": 8000, "pieces": 1 }, { "shape": "RD", "color": "FY", "clarity": "VS2", "rough_ct": 24.45, "yield_pct": 24, "price_per_ct": 5500, "pieces": 1 } ] }
To agree with the app team: final field names, pagination, error format, and webhook payloads. The contract above is the ERP-proposed baseline.
