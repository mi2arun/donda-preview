1. Purpose & Scope

This document specifies the complete functionality for the Donda ERP covering the expo bidding backend and the Rough Diamond Purchase, Sales & Logistics flow. It consolidates the original business-understanding document, the client mockup feedback (late May – early June 2026), and the decision that the ERP will act as the system of record for an externally-built expo bidding mobile app.

In Scope (Current Phase)

  • Bidding data backend & API — the ERP stores all expo data (expo/tender, lot catalog ~600 lots, bids, wins, payments) and exposes a defined API so the bidding mobile app can store & retrieve it API
  • Won Lots — derived from bid wins coming through the API (manual entry also supported)
  • Inbound logistics — shipments from expo to the buying company
  • Purchase invoice (import) with expense allocation and auto-created rough lot inventory
  • Rough Lot Master with group-level auto numbering and sub-lots
  • Sales invoice — third-party sale or sister-company transfer, with sub-lot splitting
  • Inter-company transfer — seller's price becomes buyer's cost
  • Outbound logistics and auto accounting entries

Out of Scope (This Phase)

  • The bidding mobile app UI itself — built by a separate team; the ERP only provides the data/API
  • Manufacturing / polishing module
  • Consignment & Memo rough trading FUTURE
  • India local sales GST / HSN tax computation FUTURE
  • Live courier API integration (manual tracking entry for now)

3. System Architecture & Bidding-App Integration API

A separate team is building the expo bidding mobile app. It is a front-end only — the Donda ERP is the single backend and system of record for all bidding and post-bidding data. The app reads master data from the ERP and writes all expo activity (catalog, bids, wins, payments) back to the ERP through a defined API.

┌───────────────────────────┐ ┌──────────────────────────────────────────────┐ │ EXPO BIDDING MOBILE APP │ HTTPS / REST · JSON │ DONDA ERP │ │ (built by other team) │ ─────────────────────▶ │ (System of Record) │ │ │ │ │ │ • Browse expo lot catalog│ reads masters │ Masters: companies, suppliers, expos, mines, │ │ • Inspect & submit bids │ ◀───────────────────── │ currencies, HS codes │ │ • See win/loss results │ │ │ │ • Record on-spot payment │ writes bidding data │ Bidding store: Expo · Lots · Bids · Wins · │ │ │ ─────────────────────▶ │ Payments │ └───────────────────────────┘ │ │ │ │ ▼ (a WON lot flows downstream) │ │ Won Lots ▶ Inbound ▶ Purchase ▶ Rough Stock ▶ │ │ Sales / Transfer ▶ Outbound ▶ Accounting │ └──────────────────────────────────────────────┘
Boundary decision (confirmed): The ERP stores full expo data — the bidding app does not keep its own database; everything (catalog, bids, results, payments) persists in the ERP. The ERP team defines the API contract; the app team integrates against it.

Integration Principles

  • Single source of truth — no duplicated data store on the app side; the app is online-first and reads/writes the ERP.
  • Authentication — the app authenticates as a service client (OAuth2 client-credentials or signed API key); each bidder user is identified within the request.
  • Versioned API — base path /api/v1/…; backward-compatible evolution.
  • Idempotency — write endpoints accept an Idempotency-Key so flaky mobile connections don't create duplicates.
  • Offline tolerance — the app may queue actions offline and replay them; idempotency + server-assigned IDs keep this safe.
  • Webhooks / polling — the ERP can notify the app of status changes (e.g., a won lot becomes Invoiced), or the app polls updated_since.

7. Integration API Specification API · ERP-defined

REST/JSON over HTTPS, base path /api/v1. All requests authenticated; all write requests accept an Idempotency-Key header. Timestamps are ISO-8601 UTC. This is the proposed contract — to be finalised with the app team.

7.1 Read — Master Data (app pulls)

MethodEndpointPurpose

GET/suppliersExpo/tender & third-party suppliers
GET/exposExpos/tenders (filter by status)
GET/mines · /originsMines & country origins

7.2 Write/Read — Bidding Data (app pushes, ERP stores)

MethodEndpointPurpose
POST/exposCreate an expo/tender
POST/expos/{id}/lotsUpload lot catalog (bulk, ~600 lots)
GET/expos/{id}/lotsRetrieve catalog (with updated_since)

PATCH/bids/{id}Update bid (withdraw / result)
POST/lots/{id}/resultMark Won/Lost (Won → creates Won Lot)

GET/won-lotsApp reads won lots & downstream status

7.3 Example — Submit a Bid

POST /api/v1/bids Authorization: Bearer <token> Idempotency-Key: 7c0e1f8a-... { "expo_lot_id": 4821, "bidder_user": "rahul", "bid_price_per_carat": 651.81, "bid_datetime": "2026-03-12T09:14:00Z", "remarks": "good colour, slightly included" } → 201 Created { "bid_id": 99213, "bid_status": "Submitted", "expo_lot_id": 4821 }

11. Rough Lot Master & Numbering KEY CHANGES

The Rough Lot is the core inventory unit. One rough lot is auto-created per purchase invoice line on confirmation. Lots can later be split into sub-lots at the point of sale.

11.1 Numbering Pattern

Lots are numbered {sequence}-{year}, e.g. 1-26 = lot 1 of 2026. Sub-lots append a letter:

NumberMeaning
1-26Parent rough lot, sequence 1, year 2026
1-26-AFirst sub-lot split from 1-26
1-26-BSecond sub-lot split from 1-26

11.2 Numbering Rules

  • NEW Auto-generated at Donda group level — one shared sequence across all group companies (unique group-wide).
  • NEW Manual override allowed — user can type a lot number (warn on duplicates).
  • NEW Third-party rough purchase → lot number entered manually.
  • NEW Sub-lots created during a Sales Invoice (§12); suffix auto-assigned per parent.

13. Inter-Company Transfer POINT 6

When one Donda group company transfers rough to a sister company, a single action drives both sides.

COMPANY A (Seller) COMPANY B (Buyer / Sister) ────────────────── ────────────────────────── Sales Invoice (Type = Inter-Company) ─────▶ Purchase Invoice (auto-created) • Rough Lot 1-26 / sub-lot 1-26-A • Supplier = Company A • Sale Price/Ct = X ════════════▶ • Purchase Price/Ct = X (= seller's price) • Outbound Shipment • New Rough Lot in B's stock cost (inventory value) = X (+ any B-side expenses) • Source lineage = A's 1-26-A
Rule (point 6): The selling company's sale price becomes the buying company's purchase price / cost. The transfer auto-creates the purchase entry and a new rough lot in the receiving company's stock, valued at the seller's price.
  • The sold lot/sub-lot in Company A is marked Transferred (balance reduced).
  • Company B receives a new Rough Lot (lineage back to A's lot) at cost = A's selling price + any B-side inbound expenses.
  • Outbound shipment (A) and inbound receipt (B) track the physical movement.
  • Both sides post their own accounting entries (sale in A, purchase in B).
To confirm on call: Does B get a new group lot number or keep the source number for traceability? Are inter-company margins eliminated in a group/consolidated view?

15. Accounting Integration

On Purchase Invoice Confirm

  • Debit: Rough Diamond Purchase / Inventory (goods value)
  • Debit: Expense accounts (freight, duty, insurance…)
  • Credit: Supplier / Accounts Payable

Inventory added to Rough Stock at Purchase Value + Allocated Expenses.

On Sales Invoice Confirm

  • Debit: Customer / Accounts Receivable
  • Credit: Diamond Sales
  • Debit: Cost of Goods Sold (inventory value of carats sold)
  • Credit: Rough Stock / Inventory
For inter-company transfers, Company A posts the sale set and Company B posts the purchase set; the linking goods value is the agreed transfer price (point 6).

18. Future Scope

Consignment / Memo trading. Not required now — Hong Kong buys rough outright only. Mumbai / Surat occasionally issue a rough memo for local rough sale. Later: memo issue/return, ownership-retained-until-sold tracking, memo-to-invoice conversion.
